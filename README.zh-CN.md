# agri-orchestrator 中文使用说明

这是一份给普通用户看的说明，不要求你会写代码，也不要求你了解插件内部结构。你只需要会在 OpenClaw 里用自然语言聊天，就可以开始使用这个农业记录插件。

## 这个插件能做什么

它可以帮你把种植过程中的关键信息记录下来，主要包括：

- 建立种植档案，比如花盆、地块、温室、苗床
- 记录作物信息，比如作物名称、品种、当前阶段
- 记录观察信息，比如土壤湿度、长势、病虫害、花况
- 记录操作信息，比如浇水、施肥、喷药、采收
- 做简单的种植检查，提醒你哪些信息还没补齐

目前比较适合做 MVP 级别的日常记录和测试，不适合作为正式农业合规系统的唯一依据。

## 你适合怎么用它

最简单的用法，不是直接填 JSON，也不是手动调工具，而是直接在 OpenClaw 对话框里像聊天一样说话。

例如你可以直接说：

- 帮我新建一个风信子盆栽档案，名称叫阳台风信子 1 号，类型是 container，位置是东阳台，作物是 hyacinth，品种是 Delft Blue，当前阶段 flowering。
- 帮我记录一下今天给阳台风信子 1 号浇水了，少量浇透。
- 帮我做一次当前工作区的农业检查，看看缺什么信息。

OpenClaw 会自动调用这个插件背后的工具来完成记录。

## 使用前你需要知道的 3 件事

### 1. 数据会保存到哪里

插件会把数据写到当前 OpenClaw 工作区里，通常位置是：

```text
<你的工作区>/.agri-orchestrator/data/store.json
<你的工作区>/memory/agri/*.md
```

简单理解：

- `store.json` 是主数据文件
- `memory/agri/*.md` 是便于查看的文字摘要

### 2. 它最适合记录什么

目前最适合这几类对象：

- `container`：盆栽、花盆、种植箱
- `field`：地块
- `greenhouse`：温室
- `orchard`：果园
- `nursery`：苗床、苗圃

### 3. 它不是“全自动猜测系统”

尤其是喷药、采收这类高风险记录，插件会故意严格一点：

- 如果信息不完整，会拒绝入库
- 如果内容明显像瞎猜的、占位的、用日期乱填的，也会拒绝入库

这是为了避免把错误记录写进正式数据里。

## 第一次使用：推荐顺序

如果你第一次用，建议按下面顺序来：

### 第一步：先建“种植单位”

也就是先建一个对象，比如花盆、地块、温室。

你可以这样说：

```text
请帮我新建一个种植档案，名称是风信子测试盆 A，类型是 container，位置是客厅窗边。
```

### 第二步：再补“作物计划”

也就是告诉系统，这个单位里种的是什么。

你可以这样说：

```text
请给风信子测试盆 A 添加作物信息：作物 hyacinth，品种 Delft Blue，当前阶段 flowering。
```

### 第三步：记录观察或操作

比如：

```text
帮我记录观察：风信子测试盆 A 今天花朵正常，叶片挺立，盆土微湿。
```

或者：

```text
帮我记录今天给风信子测试盆 A 浇水，少量，已确认执行。
```

## 日常怎么说，系统最容易听懂

为了让 OpenClaw 更稳定地调用插件，建议你描述时尽量带上下面几项：

- 对象名称：例如“风信子测试盆 A”
- 时间：例如“今天上午 9 点”或“2026-03-18”
- 动作：例如“浇水”“喷药”“观察”
- 具体内容：例如“少量浇透”“叶片发黄”“已见蚜虫”

### 好的说法

- 今天上午我给风信子测试盆 A 浇了少量水，请帮我记录。
- 请记录观察：风信子测试盆 A 花朵正常，叶片略微下垂，盆土偏干。
- 请为风信子测试盆 A 记录喷药，产品名是 Acme Shield，适用作物是 hyacinth，批号是 LOT-20260318-001，安全间隔期 phiDays 是 7。

### 不推荐的说法

- 你自己看着记一下吧。
- 随便补一条喷药记录。
- 就写今天喷过药了，不用细节。

这类说法太模糊，尤其在喷药场景下，大概率会被拦截。

## 常见操作示例

## 1. 新建一个盆栽档案

```text
请帮我新建一个风信子盆栽档案，名称是客厅风信子 1 号，类型是 container，位置是南窗边，作物是 hyacinth，品种是 Carnegie，当前阶段 flowering。
```

## 2. 记录一次观察

```text
请帮我记录观察：客厅风信子 1 号今天花朵开放正常，叶片无明显病斑，盆土表层微干。
```

## 3. 记录一次浇水

```text
请帮我记录今天给客厅风信子 1 号浇水，少量浇透，已经执行完成。
```

## 4. 做一次整体检查

```text
请帮我对当前工作区做一次农业检查，并用通俗中文告诉我还缺哪些重要信息。
```

## 5. 记录喷药

喷药是重点场景。你需要尽量一次性提供完整信息。

推荐这样说：

```text
请帮我记录喷药：对象是客厅风信子 1 号，时间是 2026-03-18 09:00，产品名是 Acme Shield，适用作物是 hyacinth，批号是 LOT-20260318-001，安全间隔期 phiDays 是 7，已经确认执行。
```

## 喷药为什么经常会被拦？

因为这个插件现在对喷药记录有最低合规要求。至少要有：

- `productName`：产品名
- `labelTargetCrop`：标签适用作物
- `lotNumber`：批号
- `phiDays`：采前间隔天数
- `confirmed=true`：明确确认这件事已经发生

如果你只说：

```text
今天我喷药了，帮我记一下。
```

系统通常会拒绝入库。

如果你填的是下面这种“明显不靠谱”的值，也会拒绝：

- 产品名写成作物名，比如 `hyacinth`
- 产品名写成品种名，比如 `Delft Blue`
- 产品名写成花盆名
- 产品名写成 `spray`、`pesticide`、`routine-spray`
- 批号只写一个字母，比如 `K`
- 批号直接写日期，比如 `2026-03-18`

这不是插件坏了，而是故意这样设计的。

## 什么时候建议你补充更多细节

以下情况建议尽量多说一点：

- 你要记录喷药
- 你要记录采收
- 同一个工作区里有多个同名或类似名称的对象
- 同一天做了多次操作
- 你准备把记录拿去做后续追溯

## 如何判断记录有没有真的写进去

你可以直接问 OpenClaw：

```text
请帮我确认刚才那条记录是否已经成功入库，并告诉我对象名称、时间和记录类型。
```

也可以让它帮你总结：

```text
请列出今天和客厅风信子 1 号有关的所有记录。
```

如果是技术用户，也可以直接查看：

```text
<工作区>/.agri-orchestrator/data/store.json
```

## 目前支持哪些作物

当前插件内置了以下作物规则包：

- `hyacinth`
- `corn`
- `wheat`

如果你使用的是其他作物，插件通常仍能记录数据，但在建议和检查上会更保守。

## 玉米田天气预警怎么用

如果你是把这个项目接成后端服务来用，现在已经支持“天气预报 -> 玉米田预警”的一条最小链路：

1. 给地块补上经纬度
2. 调用后端 `POST /units/:id/background/weather/forecast`
3. 再调用 `POST /plugin/units/:id/care-check`

当系统识别到：

- 当前作物是 `corn`
- 当前阶段是 `vegetative / tasseling / silking / grain_fill / flowering` 等敏感阶段
- 明天有较高降雨风险

它会生成一条针对玉米田的降雨预警，并附带一个 `corn-post-rain-scout` 提醒，方便你在雨后安排巡田。

如果你准备接飞书，建议把“同步天气”和“做一次农田检查”做成两个固定指令，连续调用会最稳。

## 原生插件里的明日天气背景

现在原生 OpenClaw 插件也支持把“明日天气摘要”写入工作区，并在构建 prompt 时自动注入背景：

1. 有地块位置时，插件会优先复用缓存中的明日天气
2. 缓存过期时，prompt hook 会自动刷新一次天气摘要
3. 大模型回答“今天有什么风险”这类问题时，就能直接参考明日天气背景
4. 如果地块只有中文位置文本，系统会先尝试自动解析并回填坐标；仍失败时，会提示你补充更标准地址或经纬度

如果你想每天定时更新一次，可以在部署机上跑：

```bash
npm run build
npm run weather:sync -- /path/to/openclaw-workspace
```

典型 cron 示例：

```cron
0 18 * * * cd /path/to/agri-orchestrator-main && npm run build >/tmp/agri-build.log 2>&1 && npm run weather:sync -- /path/to/openclaw-workspace >/tmp/agri-weather-sync.log 2>&1
```

如果地块已经记录了 `latitude` / `longitude`，会直接用坐标；否则会根据 `location` 文本做一次地名解析。

## 常见问题

### 1. 为什么我明明说了“帮我记一下”，它还是没记？

因为你的信息可能不完整，或者表达太模糊。高风险操作不会靠猜来补。

### 2. 为什么喷药记录最严格？

因为喷药涉及合规、追溯和安全问题。现在这个插件会尽量避免把“猜出来的数据”写进库里。

### 3. 我需要自己去调工具吗？

一般不需要。普通用户直接聊天就行。

### 4. 如果名字取错了怎么办？

你可以直接说：

```text
请把客厅风信子 1 号的位置改成南窗左侧。
```

或者：

```text
请帮我确认“客厅风信子 1 号”和“客厅风信子一号”是不是同一个对象。
```

## 给普通用户的最佳实践

- 名称尽量固定，不要今天叫“A 盆”，明天叫“窗边那盆”
- 时间尽量说清楚，至少写“今天”或具体日期
- 喷药时一次性提供完整字段
- 观察记录可以写得自然一点，但最好有对象名
- 同一个对象长期使用同一种叫法

## 一句话总结

你可以把它理解成一个“会聊天的种植记录员”：

- 你负责把事实讲清楚
- 它负责帮你落库、整理和检查
- 遇到喷药、采收这类高风险记录，它会宁可拦下来，也不会乱填
